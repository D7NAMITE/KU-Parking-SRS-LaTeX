\section{Software Solution}
\label{section:software-solution}
Following an iterative development process, we have successfully developed a
system consisting of a mobile application, a parking facilitator web application,
a detection model, and the supporting infrastructure. Each component serves a
distinct purpose: the mobile application assists drivers in locating vacant
parking spots, while the parking facilitator web application serves as an
administrative portal for parking area owners to manage and tune their specific
locations within the system.

Our repositories are available at https://github.com/ku-parking. The following sections of this chapter provide a detailed overview of the features and interfaces for both the mobile and web applications.


\subsection{Mobile Application - Map Overview}
\label{subsection:mobile-map-overview}
The Map Overview serves as the initial screen when a user enters the application. As shown in Figure \ref{fig:map-overview-screenshot}, the interface follows a typical map-navigation layout with a map serving as the primary source of information. The map displays the user's current location alongside parking area markers that indicate availability through both numerical values and color-coding. The color indicators are defined by the percentage of vacant spots relative to total capacity: Green represents vacancy of 70\% or higher, Yellow represents vacancy more than or equal 30\% but less than 70\%, and Red represents vacancy below 30\%. At the top of the screen, a search bar allows users to locate specific parking areas. At the bottom, a sheet drawer provides a list of all available areas, which users can select to view more granular details.

    \caption{Mobile application interface showing the map overview.}
\label{fig:map-overview-screenshot}

As shown in Figure \ref{fig:parking-spot-detail-screenshot}, these details include the parking area name, the current number of vacant spots, and a timestamp indicating when the information was last updated. \subsection{Mobile Application - Inaccuracy Report}
\label{subsection:mobile-inaccuracy-report}
In the event of data discrepancies, users can report incorrect information to
the parking facility by tapping the “Report Inaccurate Info” button, as shown in
Figure \ref{fig:report-clicking-screenshot}. This action opens a reporting form
that allows the user to select a specific reason for the report and provide an
optional description, as illustrated in Figure \ref{fig:report-form-screenshot}.
Upon successful submission, a confirmation message is displayed, as shown in
Figure \ref{fig:report-success-screenshot}. This feature empowers the end-user
to notify the parking area owner of potential data inaccuracies, ensuring the
system remains reliable through community-driven feedback.

\subsection{Web Application - Setup Parking Area}
\label{subsection:web-setup-parking-area}

Upon accessing the web application, the parking facilitator is presented with the initial dashboard shown in Figure \ref{fig:web-initial-view-screenshot}. The interface features a configuration panel that includes a camera source selection menu. This allows the administrator to choose the specific video feed intended for real-time occupancy analysis within their designated parking area.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{admin-web-app-screenshots/initial-view.png}
    \caption{Web-application dashboard showing the initial setup view for parking area configuration.}
\label{fig:web-initial-view-screenshot}
\end{figure}

The parking facilitator can select the desired camera feed source from the available options. As shown in Figure \ref{fig:web-selecting-source-screenshot}, the user selects 'Screen Capture' as the primary input. Once a source is selected, a preview is generated within the placeholder area, as illustrated in Figure \ref{fig:web-source-selected-screenshot}. At this stage, the user can crop the specific area for detection by clicking the 'Select Crop Region' button at the bottom of the interface.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{admin-web-app-screenshots/selecting-source.png}
    \caption{Web-application interface showing camera source selection for occupancy detection input.}
\label{fig:web-selecting-source-screenshot}
\end{figure}

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{admin-web-app-screenshots/source-selected.png}
    \caption{Web-application interface showing the preview of the selected camera source.}
\label{fig:web-source-selected-screenshot}
\end{figure}

Once the preview and cropping regions are correctly configured, the user can
click 'Capture \& Initialize' to deploy the detection model. As shown in Figure
\ref{fig:web-detection-model-result-screenshot}, the web application then
displays the initial output of the detection model, identifying the parking spot
that the model can detect.

\begin{figure}[H]
    \centering

    \caption{Web-application interface showing initial parking spot boundaries generated by the detection model.}
\label{fig:web-detection-model-result-screenshot}
\end{figure}

To manually define a new parking spot, the user can click the '+ Add Spot'
button. This action triggers an on-screen prompt, 'Click and drag to draw a new
parking spot,' which appears at the top of the preview window as shown in Figure
\ref{fig:web-add-parking-spot-before-screenshot}. The user can then click and
drag their cursor across the preview to create a new detection boundary, as
illustrated in Figure \ref{fig:web-add-parking-spot-after-screenshot}.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{admin-web-app-screenshots/add-parking-spot001.png}
    \caption{Web-application interface before adding a new parking spot boundary.}
\label{fig:web-add-parking-spot-before-screenshot}
\end{figure}

\begin{figure}[H]

    \caption{Web-application interface after adding a new parking spot boundary.}

The parking spots initially identified by the detection model can be manually refined by clicking on the green rectangles that indicate the parking spot placeholders. Once a specific spot is selected, its border changes to blue to signify it is in edit mode, as shown in Figure \ref{fig:web-detection-model-adjusting-before-screenshot}. The user can then adjust the boundaries by selecting an edge of the placeholder and dragging it to the desired position. This allows for precise calibration of the detection zones, with the final adjusted area shown in Figure \ref{fig:web-detection-model-adjusting-after-screenshot}.

Before registering a new parking lot within the system, the facilitator must
input specific details, including the name of the parking area and its
geographic coordinates. As shown in Figure \ref{fig:web-add-parking-lot-detail-screenshot}, the location is defined using the interactive map interface situated on the right side of the page. This ensures that the parking lot's physical location is accurately mapped to synchronize with the mobile application's navigation features.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{admin-web-app-screenshots/add-parking-lot-detail.png}
    \caption{Web-application interface showing parking lot detail entry, including map-based location selection.}
\label{fig:web-add-parking-lot-detail-screenshot}
\end{figure}

Upon clicking 'Save \& Start', the system officially registers the parking lot and initiates the vehicle detection model. The facilitator can then monitor the live parking availability through the dashboard interface, as shown in Figure \ref{fig:web-saved-parking-lot-screenshot}. Should the configuration require further refinement, the user can select 'Edit Spots' to adjust existing parking area placeholders or 'Reset' to completely reconfigure the detection setup from the initial stage.

\end{figure}
