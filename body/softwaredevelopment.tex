\section{Software Development Methodology}
\label{section:software-development-methodology}
This chapter describes the development lifecycle, technical stack, and
implementation strategies for the KU Parking system. It details how the project
transitions from requirements to a functional solution.

\subsection{Methodology}
\label{subsection:methodology}
The project utilizes an Agile-bassed framework slightly modified to a two-person team.
Our primary goal is to maintain a balance between driving development forward and 
allowing a flexiibility to adapt to possible changes or technical challenges. 
The development process is structured around weekly sprints, in which the team develops in parallel on both 
the AI component and the user's side application. 
This approach allows us to maintain a clear focus on both the technical implementation, ensuring a feedback
from advisors and stakeholders cann be reconsidered and applied in a timely manner.
\linebreak 
The key adaptations include:
\begin{itemize}
    \item \textbf{Role Separation:} The development workload is split into two primary tracks 
    to ensure specialized focus:
    \begin{itemize}
        \item \textbf{AI Component:} Responsible for the YOLO-based
        vehicle detection, edge detection for parking boundaries, and the
        IoU-based occupancy algorithm.
        \item \textbf{Software Architecture and Integration:} Responsible for the mobile interface, 
        backend service server, and user-end features.
    \end{itemize}
    \item \textbf{Weekly Meetings:}  In place of daily stand-ups, 
    the team meets once per week to synchronize progress, 
    address blockers, and update the work plan for the following week.
    \item \textbf{Constant Communication Between Members:} Regular communication within the teammembers are constantly maintained, 
    unlike the traditional Agile approach. This is possible due to the small team size, allowing for 
    quick decision-making and problem-solving without the need for formal meetings.
\end{itemize}

\subsection{Technology Stack}
\label{subsection:technology-stack}
The system consists of several components, each utilizing specific technologies to meet the project requirements:
\begin{itemize}
    \item \textbf{AI Component:}
    \begin{itemize}
        \item \textbf{YOLO:} For real-time vehicle detection in parking lot images.
        \item \textbf{Python:} The primary programming language for AI development, 
        leveraging libraries such as PyTorch for model training and inference.
    \end{itemize}
    \item \textbf{Mobile Application:}
    \begin{itemize}
        \item \textbf{Expo:} To develop a cross-platform mobile application for both Android and iOS.
        \item \textbf{React Native:} The framework used within Expo for building the user 
        interface and handling application logic.

    \end{itemize}
\end{itemize}

\section{Coding Standards}
\label{section:coding-standards}

\subsection{Mobile Application (TypeScript)}
\begin{itemize}
    \item \textbf{Formatting and Linting:} We use \textbf{Prettier} for automatic code formatting and \textbf{ESLint} for static analysis. This ensures a consistent style, such as the use of single quotes, 2-space indentation, and mandatory semi-colons.
    \item \textbf{Naming Conventions:} 
    \begin{itemize}
        \item \textbf{PascalCase} for component filenames and React component.
        \item \textbf{camelCase} for variables, functions, and custom hooks.
        \item \textbf{UPPER\_SNAKE\_CASE} for global constants.
    \end{itemize}
    \item \textbf{Functional Components:} All UI elements must be built using functional components and React Hooks.
\end{itemize}

\subsection{AI Component (Python)}
\begin{itemize}
    \item \textbf{Style Guide:} Adherence to \textbf{PEP 8} standards for all Python code.
    \item \textbf{Documentation:} All detection functions must include Sphinx-style docstrings to explain parameters and the underlying mathematical logic (e.g., IoU calculations).
    \item \textbf{Naming Conventions:} \textbf{snake\_case} for functions and variables; 
    \textbf{PascalCase} for classes.
\end{itemize}
